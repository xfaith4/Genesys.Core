System Overview
End-to-end view of the three-module ecosystem. Callers authenticate via Genesys.Auth, invoke datasets through Genesys.Core, and receive results either directly or through the Genesys.Ops high-level cmdlets. Each path flows through catalog resolution, paging dispatch, HTTP retries, field redaction, and JSONL output.

Module Structure
Three PowerShell modules with clear separation of concerns. Genesys.Auth handles token lifecycle. Genesys.Core is the data-extraction engine with 3 public and ~25 private functions. Genesys.Ops wraps Core with 100+ domain-specific cmdlets.

Paging Strategies
Invoke-CoreEndpoint switches on the paging.profile field from the catalog endpoint spec. Each strategy handles a different Genesys Cloud API convention. All strategies share the same HTTP + retry layer underneath.
Profile Resolution: The paging.profile key in the catalog resolves to a named profile object (e.g. nextUri) which provides runtime settings like nextUriPath, pageParam, maxPages, and itemsPath.

Retry & HTTP Layer
Every paging iteration goes through Invoke-RequestWithRetry → Invoke-WithRetry → Invoke-GcRequest → Invoke-RestMethod. The retry engine applies exponential backoff with jitter and honours Retry-After headers. URI query parameters containing secrets are redacted before logging.
Backoff formula: delay = Min(baseDelay × 2^attempt + Random(0, jitter), maxDelay) · Default: base=1s · max=60s · jitter=0–0.25s · maxRetries=3 · retryOn=[429]

Async Job Flow
Used for audit-logs and analytics-conversation-details. The Genesys Cloud API requires a submit → poll → fetch results pattern. Invoke-AsyncJob orchestrates all three phases, substituting the jobId into result endpoint path templates via Resolve-AsyncJobRouteValues.

Authentication Flows
Genesys.Auth provides three OAuth2 flows. All three store the resulting Authorization: Bearer … header in module-scoped state and optionally in a DPAPI-encrypted file on disk. Downstream callers pass the stored headers directly to Invoke-Dataset.

Run Output Structure
Every dataset run creates a self-contained folder under out/<datasetKey>/<runId>/. The runId is an ISO 8601 UTC timestamp. Four files are produced: a streaming data JSONL, an append-only events log, a final manifest, and an optional summary.

Catalog & Configuration
The catalog (catalog/genesys.catalog.json) is the single source of truth for API metadata. It defines datasets, endpoints, and named profiles for paging and retry. All runtime settings are derived from it at execution time.
